# Serial backend — design

> **Date:** 2026-06-21
> **Status:** approved (brainstorming) → pending implementation plan
> **Plan:** _to be created via writing-plans_

## Context & purpose

Mesh Tester today injects **virtual** Meshtastic nodes over MQTT (zero hardware).
The project's actual goal is broader: a **web tool that manages a fleet of test
nodes — each backed by either a real device connected over serial (USB) or a
virtual MQTT-injected node — used together to test *other* nodes** (the mesh
under examination).

This spec adds the serial path as a first-class, selectable backend that
coexists and mixes with the virtual one.

## Decisions (from brainstorming)

| Question | Decision |
|----------|----------|
| Serial vs MQTT relationship | **Additional backend (toggle)** — MQTT virtual injection stays; serial is a selectable backend. Real and virtual nodes coexist in one mixed fleet. |
| Serial scope | **Full control panel** — TX + RX + device admin (admin deferred to Phase 2). |
| What "test other nodes" returns | **Traffic generation + complete, filterable feed** (TX + RX). No formal delivery/ACK/RSSI report subsystem for now; signal fields present in RX packets are surfaced in the feed. |
| Architecture | **A — Unified `NodeBackend` abstraction.** |

## Goals

- A common `NodeBackend` interface with two implementations: virtual (MQTT) and serial (real device).
- Connect / disconnect one or more real nodes by serial port at runtime.
- Send `text` / `position` / `telemetry` from any node (real or virtual), manually or via scenarios.
- A unified, client-side **filterable feed** of TX and RX events (filter by node, type, direction, channel).
- Refactor the existing MQTT path behind the abstraction **without breaking current tests**.

## Non-goals (YAGNI / deferred)

- Device admin (set owner/channel), traceroute, nodelist import → **Phase 2**.
- Scenario auto-traffic driving *serial* nodes → **Phase 2** (Phase 1: serial nodes send manually).
- Structured delivery verification / ACK tracking / range-quality statistics.
- Running serial inside the Docker container (USB passthrough) — documented caveat, host-run instead.

## Architecture

### `NodeBackend` interface

```
id: str                 # "!xxxxxxxx"
longname: str
kind: str               # "virtual" | "serial"
connected: bool
auto_traffic: bool      # participates in scenario scheduler (Phase 1: serial=False)

connect() -> None
disconnect() -> None
send_text(text: str, to: str | None = None, channel: int = 0) -> None
send_position() -> None
send_telemetry(metrics: dict) -> None
# RX: only serial backends emit; they push events into a shared sink (see RX bridge)
```

### Implementations

- **`VirtualBackend`** — wraps the existing `VirtualNode` + `MqttInjector` (the
  current `NodeInjector` logic moves here). `send_*` build the existing JSON
  payloads and publish to MQTT. No RX.
- **`SerialBackend`** — wraps `meshtastic.serial_interface.SerialInterface(devPath=port)`.
  `id`/`longname` read from `getMyNodeInfo()`. `send_text` → `iface.sendText`,
  `send_position` → `iface.sendPosition`, telemetry via the lib's telemetry
  helper (exact call confirmed against the installed version at implementation
  time). RX via `pubsub.subscribe(cb, "meshtastic.receive")`.

### State

`MeshState.backends: list[NodeBackend]` — a **mixed** list. Virtual nodes are
still generated by `NodeFactory`; serial nodes are added at runtime when a port
is connected. `_emit` is generalized to call `backend.send_*`, then log the TX
event to the feed + `Recorder` + WebSocket broadcast.

### RX bridge (thread → asyncio)

`SerialInterface` runs a synchronous reader thread and dispatches via `pubsub`.
At startup the app captures the running event loop. A `SerialBackend`'s pubsub
callback (running on the reader thread) uses `loop.call_soon_threadsafe` to
hand the received packet to `ConnectionManager.broadcast` as an `rx` feed event.
The asyncio loop is never blocked by serial I/O.

## API surface (Phase 1)

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/serial/ports` | list candidate serial ports (`meshtastic.util.findPorts` / pyserial) |
| `POST` | `/api/serial/connect` | body `{port}` → create `SerialBackend`, return node info |
| `POST` | `/api/serial/{id}/disconnect` | close + remove a serial backend |
| `POST` | `/api/nodes/{id}/send` | body `{type, ...}` → manual send from any node (real or virtual) |

Existing endpoints (`/api/status`, `/api/nodes`, `/api/scenario/*`,
`/api/zone/*`, start/stop/pause) stay; `/api/nodes` now includes `kind` and
`port` per node.

## WebSocket / feed

New event `{"type":"log","level":"rx", from, node_type, ptype, snr, rssi, hops, payload, ts}`.
Existing `log` levels (`text`/`position`/`telemetry`) become **TX** entries.
The dashboard renders a unified feed; filtering (by node, type, direction TX/RX,
channel) is done client-side over the `log` array.

## UI

- Node list: mixed nodes with a **VIRT** / **SERIAL · `<port>`** badge.
- "Connect serial device" control with a port dropdown (from `/api/serial/ports`).
- Per-node manual send panel (text / position / telemetry).
- Feed panel: complete TX + RX feed with filter chips.

## Data flow

- **TX:** UI or scenario → `backend.send_*` → (virtual: MQTT publish · serial:
  LoRa radio TX) → feed + `Recorder` + WS broadcast.
- **RX:** radio → pubsub callback (reader thread) → `call_soon_threadsafe` →
  `rx` feed event → WS broadcast. Virtual backends produce no RX.

## Error handling

- Port missing / busy / device unplugged mid-session / reader-thread exception →
  backend marked `disconnected`, error surfaced in UI, **app does not crash**.
- `meshtastic` / pyserial missing, or no device present → serial endpoints
  return a clean error, not a 500 stack trace.
- Reconnect is manual (Phase 1).

## Deployment caveat

USB serial does **not** work inside the current Docker container (no
`/dev/ttyUSB*`). Serial mode runs on the **host** (`python web_main.py`), or via
compose with explicit `devices:`/group mapping. To be documented in the README.

## Testing

- `SerialBackend` unit-tested with a **fake `SerialInterface`** (mock the
  meshtastic lib — no hardware in CI): asserts `send_*` map to the right iface
  calls, and an injected RX callback produces the expected feed event.
- API tests for `/api/serial/ports`, connect/disconnect (mocked), manual send,
  and the mixed node list.
- The existing MQTT/web/TUI test suite must stay green after the refactor.

## Phasing

- **Phase 1 (MVP):** `NodeBackend` abstraction + refactor MQTT into
  `VirtualBackend`; serial connect/disconnect + port list; serial TX
  (text/position/telemetry); RX feed with filters; mixed node list UI.
- **Phase 2:** device admin (owner/channel), traceroute, nodelist import,
  scenario auto-traffic on serial nodes, ACK/delivery evidence.

## To confirm at implementation time (not assumed)

- Exact `meshtastic` API: `sendPosition` / telemetry helper signatures, RX
  packet shape (`decoded`, `rxSnr`, `rxRssi`, `hopLimit`), and admin calls —
  verified against the installed `meshtastic` version, not from memory.
